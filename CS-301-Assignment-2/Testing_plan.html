Given this is a more complicated algorithm, I went ahead and tested extensively

Unit tests are given for almost every function within the implementation of the Trotter algorithm

These were built upon, first checking that we could generate a set of numbers

Then that we’d find the correct largest mobile

Then that this mobile was swapped correctly

Then that all the larger elements than that were reversed in direction

Then that a single iteration of the algorithm would execute correctly on a small set of test cases

Lastly, the functionality that combines all these together was tested extensively

It tests that the correct number of permutations are generated, 

that these permutations are individually unique,

And that these permutations contain nothing but the originally generated elements

The submitted project listing contains all permutations 

[ [1, 2, 3],

  [1, 2, 3, 4],

  [1, 2, …, 5],

  [1, 2, …, 6],

  [1, 2, …, 7],

  [1, 2, …, 8] ]


It can happily go higher, but, at this point we’re generating 5000 some permutations, and the 

Project listing would likely balloon in size were we to go higher and include these

Each checks out as specified and as can be seen in the assertions written in the unit tests
